	Frontend Flutter Developer
Objetivo
Desarrollar una aplicación móvil en Flutter que consuma la API pública de Catfact Ninja para crear
un directorio interactivo de razas de gatos. Se evaluará la arquitectura del código, el manejo de
estados, el consumo eficiente de la API (incluyendo paginación) y las buenas prácticas de UI/UX ante
posibles fallos de red.
Se tomará en cuenta la creatividad y solución propuesta ante situaciones de posibles fallos que
deben ser manejadas para garantizar la correcta experiencia del usuario.
Getting Started
Crear el proyecto Flutter con el nombre cat-directory-app.
Hacer un commit inicial con la estructura base.
PLUS: proveer una versión compilada (APK o Web build) en los releases del repositorio.
Contexto del Negocio
Eres el Lead Mobile Developer para una nueva iniciativa orientada al cuidado de mascotas. Necesitan
un MVP (Minimum Viable Product) de una aplicación que:
1. Permita a los usuarios explorar un directorio de razas de gatos.
2. Visualice los resultados de manera fluida, manejando grandes volúmenes de datos sin
comprometer el rendimiento.
3. Ofrezca detalles específicos de cada raza y un dato curioso aleatorio.

en una lista (ListView o SliverList). (documentación de la api https://catfact.ninja/ )
- Información a mostrar: Nombre de la raza (breed) y el país de origen (country).

Al revisar el repositorio del candidato, enfócate en estos 4 pilares. Esto te dirá rápidamente dónde

1. Manejo de Estado (BLoC) y Paginación (El Filtro Principal)
Esta es la prueba de fuego. Revisa el archivo del BLoC (ej. breeds_bloc.dart).
Bandera Roja (Junior / Copy-Paste): Toda la lógica de la API está metida dentro del archivo
de la interfaz (UI) usando un setState. Si usaron BLoC, cada vez que hacen scroll sobrescriben
la lista completa (la pantalla hace saltos raros). No hay control de eventos: si haces scroll
rápido, la consola muestra 10 llamadas a la API al mismo tiempo.
Aceptable (Mid-Level): Entienden que BLoC separa la lógica. Emiten estados de carga y
logran anexar los datos nuevos a la lista existente (nueva_lista = [...lista_vieja,
...datos_nuevos]).
Excelente (Senior): Implementan "transformación de eventos". Usan utilidades como
droppable (ignorar eventos de scroll mientras ya hay una petición en curso) o debounce para
no saturar la API. Mantienen un estado inmutable y limpio.
2. Consumo de API y Capa de Datos
Busca cómo se conectan a https://catfact.ninja.
Bandera Roja (Junior): Las URLs están "quemadas" (hardcoded) directamente en los
botones o en la UI. No usan modelos de datos, sino que trabajan directamente con
Map<String, dynamic>.
Aceptable (Mid-Level): Crearon una clase BreedModel con su método fromJson. Tienen un
archivo separado (ej. api_service.dart o breed_repository.dart) que hace las peticiones
HTTP.
Excelente (Senior): Centralizan la configuración de la API (un cliente HTTP base). Manejan
excepciones a nivel de repositorio y retornan respuestas tipadas usando clases envolventes
como Either (del paquete fpdart o dartz) o Result (Éxito / Fracaso) para que el BLoC decida
qué hacer.
3. UI, UX y Manejo de Errores
Ejecuta la app (o revisa su código de UI) e imagina que estás en el metro perdiendo la señal de Digitel
o Movistar.
Bandera Roja (Junior): Si apagas el WiFi y abres la app, se queda una pantalla gris infinita o
explota con la famosa "pantalla roja de la muerte" de Flutter.
Aceptable (Mid-Level): Si hay un error, muestran un texto simple o un Snackbar genérico.
Tienen un indicador de carga (CircularProgressIndicator) básico en el centro de la pantalla.
Excelente (Senior): Tienen Skeleton Loaders (placeholders grises que brillan) mientras carga
por primera vez. Si el Infinite Scroll falla en la página 3 por falta de red, no borran las páginas
1 y 2; simplemente muestran un botón de "Reintentar" al final de la lista.
